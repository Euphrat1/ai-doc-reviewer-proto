## Future / backlog

### 1) Точный подсчёт токенов “как у модели”
**Связано с ТЗ:** `tech_req.md`, раздел **F9. Расчёт токенов (по файлам и по промпту)**.

**Проблема:** сейчас используется базовая оценка токенов по символам (estimate). Она полезна для UX, но может ошибаться на:
- русском тексте,
- таблицах/CSV,
- коде, идентификаторах, e-mail/телефонах, путях,
- “шумном” OCR.

**Фича:** добавить опциональный режим точного токен-счёта под выбранную модель:
- если выбранная модель поддерживается токенайзером — считать токены максимально близко к реальному API;
- если нет — падать обратно на базовую оценку и явно маркировать это в UI.

**Требования к UX:**
- переключатель: `Token counting mode: basic / model-accurate`
- индикация точности: “оценка” vs “точно по токенайзеру модели”
- кэширование результатов (чтобы пересчёт не тормозил предпросмотр)

### 2) Обратная подстановка PII по JSON-маппингу
**Связано с PII сервисом (отдельный репозиторий):** контракт API (api_version=1) описан в `contracts/pii/openapi.yaml` (зеркало).

**Проблема:** после маскирования текст становится безопасным для LLM, но пользователю может понадобиться восстановить исходные данные в локальной копии результата или экспортировать две версии документа: masked и original-aware.

**Фича:** добавить локальную функцию обратной подстановки масок в исходные значения на основе `replacements` из JSON-маппинга.

**Требования к UX:**
- хранить `masked_text` и `replacements` вместе как связанную пару данных;
- добавить действие `Restore masked data locally`;
- перед восстановлением явно предупреждать пользователя, что в тексте снова появятся чувствительные данные;
- поддержать экспорт двух версий: masked и restored.

### 3) PII маскирование OCR-результатов
**Проблема:** OCR (`OCR_TEXT`, `PDF_OCR`) может содержать PII и сейчас не проходит через PII Masking Service.

**Фича:** добавить опцию “Mask OCR outputs”, которая дополнительно прогоняет OCR-результаты через PII Masking Service:
- режимы: `off` (default) / `on`
- при включении: маскировать OCR-текст отдельными вызовами сервиса или объединять с основным текстом (зависит от лимитов и качества).

**UX/безопасность:**
- предупреждать, что обработка OCR может быть медленнее;
- `replacements` остаётся локальным и экспортируется в отчёт как чувствительные данные.
