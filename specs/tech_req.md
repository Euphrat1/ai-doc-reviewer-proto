## Техническое задание (v0.3)

Версия v0.3 уточняет лимиты контекста/выхода “по умолчанию” (с настройкой из UI), добавляет расчёт токенов по каждому файлу и по итоговому промпту, а также фиксирует подход “Full Corpus → Prompt Corpus” для управления объёмом контекста.

### 1. Цель
Локальное Windows-приложение для анализа комплекта материалов с помощью LLM.

Пользователь выбирает папку с материалами. Приложение:
- Загружает и парсит материалы в текстовый корпус.
- (Опционально) выполняет OCR изображений и PDF-страниц (локально).
- Выполняет маскировку PII через локальный HTTP-сервис PII Masking Service (отдельный репозиторий; контракт интеграции см. Appendix A, api_version=1).
- Отправляет задание и материалы в OpenRouter для анализа, запрашивает оценку уверенности и (при необходимости) уточняющие вопросы.
- Позволяет отвечать на вопросы, прикладывать дополнительные файлы и выполнять финальный повторный анализ.
- Всегда даёт возможность просмотреть полный промпт перед отправкой (без редактирования, но с возможностью копировать).

Требование к реализации: код, структура проекта и использование библиотек должны быть минимальны и понятны начинающему программисту.

### 2. Функциональные требования

#### F1. Загрузка материалов
- Пользователь вводит путь к папке или выбирает через кнопку «Обзор» (системный диалог).
- Приложение рекурсивно собирает все файлы с расширениями:
  - `.txt`, `.md`, `.csv`, `.xlsx`, `.xml`, `.bpmn`, `.drawio`, `.png`, `.jpg`, `.pdf`.
- Текстовые материалы и результаты парсинга (см. F1.2) отображаются как просматриваемый текст.
- Изображения отображаются в интерфейсе как превью.

#### F1.2. Парсинг материалов в корпус для LLM (качество ответа)
Для каждого файла формируется **нормализованный текстовый блок** в двух слоях:
- **STRUCTURE**: компактная структура/метаданные, чтобы LLM быстро понимала “что внутри”.
- **EVIDENCE**: фрагменты/выборки, близкие к исходнику, чтобы LLM могла опираться на конкретные факты.

Блок каждого файла обязан начинаться с заголовка:
`FILE: <relative_path> | TYPE: <ext> | SIZE: <bytes> | PARSE: ok|warn|fail`

**Важно (вариант B):** парсинг формирует **Full Corpus** (полный корпус) — максимально полный результат парсинга (STRUCTURE+EVIDENCE) без ужатия под текущую модель/бюджет. Лимиты применяются на этапе сборки промпта (см. F1.3).

Детальные правила парсинга по типам файлов вынесены в `specs/parsing_rules.md`.

#### F1.3. Лимиты корпуса и политика сжатия (настраивается из UI)
Цель: обеспечить предсказуемое качество и избежать переполнения контекста модели.

**Определения:**
- **Full Corpus**: результат парсинга материалов (STRUCTURE+EVIDENCE) в максимально полном виде.
- **Prompt Corpus**: корпус, полученный из Full Corpus путём сжатия/семплинга под текущие лимиты и выбранную модель; именно он включается в промпт, отправляемый в OpenRouter.

**По умолчанию** (ориентация на модели с контекстом ~128k, например `openai/gpt-4o-mini`):
- **Prompt budget (цель)**: до **70,000** токенов на весь запрос (инструкции + корпус + задание), чтобы оставлять запас.
- **Лимит ответа**: `max_output_tokens` по умолчанию **4,000** (настраиваемо).
- **Мягкие лимиты на файл**:
  - STRUCTURE: до **1,000** токенов на файл.
  - EVIDENCE: до **5,000** токенов на файл.
- **PDF (по умолчанию)**:
  - максимум **12** страниц в EVIDENCE на один PDF (настраиваемо),
  - при превышении: включается выборка (первые страницы + выборка остальных).

Все лимиты обязаны быть **настраиваемыми из интерфейса** (см. UI).

**Политика сжатия Prompt Corpus при превышении prompt budget** (по умолчанию):
1. STRUCTURE сохраняется всегда (в пределах лимита STRUCTURE на файл).
2. EVIDENCE режется до лимитов: сначала по “менее приоритетным” файлам (порядок: изображения OCR → таблицы → прочее), затем по всем файлам равномерно.
3. Если всё ещё превышено: EVIDENCE оставляется только для первых N файлов (N настраиваемо), для остальных остаётся только STRUCTURE.

#### F1.3.1. Прозрачность сжатия Prompt Corpus (контракт качества)
При формировании **Prompt Corpus** из **Full Corpus** приложение обязано:
- Показывать пользователю сводку “до/после”:
  - `tokens_full_corpus_est` (суммарно),
  - `tokens_prompt_corpus_est` (суммарно),
  - `tokens_prompt_total_est` (инструкции + Prompt Corpus + задание).
- Показывать, что было сжато (в человекочитаемом виде):
  - список файлов, у которых **EVIDENCE был обрезан** (с указанием “сколько было / сколько стало” в токенах-оценках),
  - список файлов, у которых **EVIDENCE полностью удалён** (остался только STRUCTURE),
  - для `.pdf`: какие страницы попали в Prompt Corpus (например `included_pages: [1,2,5,6]`),
  - для таблиц `.csv/.xlsx`: какие строки/диапазоны вошли в выборку (например “first N + sampled M”).
- Показывать применённую политику сжатия: какой шаг сработал (резка EVIDENCE, ограничение на N файлов с EVIDENCE и т.п.).
- Предоставлять предпросмотр **Prompt Corpus** отдельно от **Full Corpus** (read-only), чтобы пользователь видел ровно то, что уйдёт в OpenRouter.

#### F2. OCR (локально, Tesseract)
OCR — опциональная функция для улучшения качества, выполняется локально.

##### F2.1. OCR для изображений `.png`, `.jpg`
- Настройка: `OCR изображений: выкл/вкл`.
- При включении EVIDENCE включает `OCR_TEXT:` (с языком и метаданными).
- OCR выполняется до маскировки PII.

##### F2.2. OCR для `.pdf`
- Настройка: `OCR PDF: выкл/вкл`.
- Алгоритм:
  - попытка извлечь текстовый слой;
  - если “текста мало” (порог в настройках), страница OCR-ится.
- EVIDENCE включает `PDF_TEXT p.<n>:` и/или `PDF_OCR p.<n>:`.

##### F2.3. Настройки OCR
- Язык OCR по умолчанию: `rus+eng` (изменяемо).
- Лимиты: максимум страниц PDF для OCR, максимум изображений для OCR (настраиваемо).
- Режим качества: `быстро/качественно`.

#### F3. Маскировка персональных данных (через локальный PII Masking Service)
Цель: перед отправкой во внешние LLM (OpenRouter) использовать только замаскированный текст.

- По кнопке «Скрыть личные данные» текстовые материалы корпуса (без OCR-результатов) отправляются в локальный HTTP-сервис **PII Masking Service** (отдельный репозиторий; контракт см. Appendix A, api_version=1).
- Приложение передаёт `trace_id` (генерируется приложением) и получает результат маскирования:
  - `masked_text` (string): текст с заменами на типизированные маски `<TYPE_N>`
  - `replacements` (array): соответствия `{ mask, type, original }` (чувствительные данные)
- В интерфейсе отображается `masked_text` (замещает текст корпуса).
- Для внешних запросов (OpenRouter) используется **только** `masked_text`.
- `replacements` хранится локально в рамках сессии и **не отправляется во внешние API**.
- После успешного маскирования приложение должно предоставить возможность **сразу просмотреть**:
  - `masked_text` (как основной результат),
  - `replacements` (как чувствительные данные) — с явным предупреждением в UI (“содержит PII”).
- Выбор sync/async:
  - если `text` <= `input_limits.max_chars` выбранного метода — использовать `POST /mask`
  - иначе — `POST /mask_async` + polling `GET /jobs/{job_id}`
- Если сервис недоступен или вернул ошибку — приложение показывает ошибку и **не отправляет unmasked текст во внешние API** (fail-closed по умолчанию).
- Исключение (явно разрешаемое пользователем): чекбокс **“Разрешить отправку без маскирования”**.
  - По умолчанию выключен.
  - Если включён, приложение может отправлять во внешние API (OpenRouter) **исходный текст** без маскирования, но обязано:
    - показать заметное предупреждение о риске утечки PII,
    - добавить отметку в лог/отчёт, что запрос был выполнен без маскирования.

Настройки интеграции (настраиваются из UI):
- `PII_SERVICE_URL`
- `PII_METHOD_ID` (по умолчанию `local_llm`)
- `PII_OPTIONS` (настраивается из UI; в MVP показывать только):
  - `model_id` (string, optional)
  - `timeout_seconds` (integer, default=90)
- `ALLOW_SEND_WITHOUT_PII_MASKING` (bool, default=false)

#### F4. Отправка задания на анализ (OpenRouter)
- Пользователь вводит текст задания.
- Автоматически добавляется системная инструкция с правилами уверенности и генерации вопросов:
  - Требуется дать развёрнутый ответ.
  - Вернуть `confidence` в процентах (1–100).
  - Если `confidence` **<50** — сформулировать **3** уточняющих вопроса.
  - Если `confidence` **<65** — **2** уточняющих вопроса.
  - Если `confidence` **<80** — **1** уточняющих вопроса.
  - Если `confidence` **≥80** — вопросов **0**.
- Перед отправкой доступен предпросмотр полного промпта.
  - Текстовые шаблоны промптов (универсальный и специализированные) описаны в файле `promts.md`.

#### F4.1. Формат ответа LLM (строго для парсинга)
LLM должна возвращать **строго JSON** (без Markdown и без текста вне JSON).
Конкретные схемы ответов (primary/manual/final) и шаблоны промптов описаны в `specs/promts.md` и являются источником правды.

Если JSON распарсить не удалось:
- 1 автоматическая попытка “исправить формат” отдельным запросом,
- затем показать сырой ответ и ошибку парсинга.

#### F4.2. Ошибки, ретраи и логирование (OpenRouter)
- **Timeout**: 90 секунд на запрос (настраиваемо из UI).
- **Retry**: до 2 повторов (итого 3 попытки) только для **429 / 5xx / timeout / сетевых ошибок**, с экспоненциальной паузой (2s, 4s).
- **Строгий JSON**: если ответ невалидный JSON → 1 попытка “reformat-to-json” отдельным запросом; если снова невалидно → показать сырой ответ + ошибку парсинга + кнопку “Повторить”.
- **Trace ID**: для каждого запроса генерируется короткий идентификатор (например `req_YYYYMMDD_HHMMSS_xxxx`). Он отображается рядом с результатом/ошибкой и записывается в лог.

**Минимальные требования к логированию (для диагностики качества и сбоев):**
- В рамках сессии приложения хранить журнал событий по запросам (без обязательной записи на диск).
- Для каждого запроса в журнале сохранять:
  - timestamp,
  - тип запроса: `mask_pii` / `analyze` / `manual_question` / `final_analyze` / `reformat_to_json`,
  - выбранную модель,
  - номер попытки и причину ретрая (если был),
  - итог: success/error, HTTP-код (если есть), краткий текст ошибки,
  - `tokens_prompt_total_est` и факт применения сжатия Prompt Corpus (да/нет).
- В UI предоставить кнопку “Показать детали” для последнего запроса (и/или скачать лог как `.json` — опционально).

**Сохранение результатов (минимум):**
- В UI предоставить кнопку “Сохранить отчёт” (экспорт в файл `.json`).
- Отчёт должен включать минимум:
  - `trace_id`,
  - timestamp,
  - выбранную модель,
  - `tokens_full_corpus_est`, `tokens_prompt_corpus_est`, `tokens_prompt_total_est`,
  - summary сжатия Prompt Corpus (какие файлы/страницы/выборки вошли, что было обрезано),
  - исходное задание,
  - итоговый ответ модели (и при наличии — список вопросов и ответы пользователя для финального анализа).
- Результаты маскировки PII включаются в отчёт **по умолчанию**:
  - `masked_text`
  - `replacements` (как чувствительные данные)
  - при экспорте показывать предупреждение, что отчёт содержит PII.

#### F5. Обработка ответа
- Показать ответ, уверенность, вопросы (если есть).
- На каждый вопрос: поле ответа + возможность прикрепить 1 файл.

#### F6. Ручной запрос уточняющих вопросов
- Кнопка доступна после загрузки материалов.
- Генерируется **1** самый полезный вопрос.

#### F7. Финальный анализ с уточнениями
- Итоговый запрос: корпус + задание + список вопросов/ответов + содержимое прикреплённых файлов.
- Итоговый ответ без генерации новых вопросов.

**Требования к блоку Q/A (для предотвращения разъезда контекста):**
- В финальный запрос должны включаться **точные строки вопросов** (без перефразирования), ровно в том виде, в каком они были получены от LLM и показаны пользователю.
- Для каждого вопроса ответ пользователя должен идти **сразу рядом** с вопросом.
- Порядок вопросов должен сохраняться.
- Если к ответу прикреплён файл, в финальный запрос добавляется **распарсенное содержимое** прикреплённого файла (или OCR-текст/выжимка — по правилам парсинга) под этим же вопросом.
- (Опционально) каждому вопросу присваивается стабильный `question_id` (`q1`, `q2`, …) для связи UI ↔ промпт, но текст вопроса всё равно передаётся полностью.

Рекомендуемый формат `Q_AND_A` (как контракт строки, включаемой в пользовательское сообщение при финальном анализе; см. `promts.md`):

Q_AND_A:
[1] QUESTION: <exact question string from LLM>
    ANSWER: <user answer text>
    ATTACHMENTS:
      - name: <filename>
        content: <parsed text or OCR text>
[2] QUESTION: ...
    ANSWER: ...

#### F8. Предпросмотр промпта
- Для маскировки и OpenRouter доступен предпросмотр полного промпта (read-only).

#### F9. Расчёт токенов (по файлам и по промпту)
Цель: сделать объём контекста прозрачным и управляемым.

После парсинга каждого файла (Full Corpus) приложение рассчитывает **оценку токенов**:
- `tokens_structure_full_est`
- `tokens_evidence_full_est`
- `tokens_total_full_est`

Требования к отображению и “прозрачности сжатия” Prompt Corpus (включая токен‑сводку до/после) — источник правды: `F1.3.1`.

Требования:
- токены считаются **оценочно** (estimate), без “настоящего” токенайзера выбранной модели;
- метод базовой оценки по умолчанию:
  - считать токены как \(tokens \approx \lceil chars / k \rceil\), где `chars` — количество символов в нормализованном тексте (включая пробелы),
  - `k` по умолчанию = **3.0** (консервативно для смешанного RU/EN),
  - в UI коэффициент `k` настраиваемый (например, 2.5–4.0) и подписан как “приблизительная оценка”;
- при смене модели или изменении лимитов/политики сжатия пересчитывается **Prompt Corpus** и токены промпта (без перепарсинга);
- если оценка превышает `prompt_budget_tokens` — UI показывает предупреждение и применённую политику сжатия;
- в UI показывается “что именно было сжато” (например: какие файлы потеряли EVIDENCE и остались только со STRUCTURE).

## Appendix A — PII Masking Service Contract (для отдельного репозитория)

Минимальные методы API: `GET /health` (проверка доступности/версии), `GET /methods` (список методов и лимитов), `POST /mask` (sync-маскирование), `POST /mask_async` + `GET /jobs/{job_id}` (async-маскирование для больших текстов).

Контракт API для **PII Masking Service** описан в OpenAPI (источник правды):
- `contracts/pii/openapi.yaml` (копия, синхронизируемая из PII Service Repo)

Синхронизация:
- обновление контракта выполняется скриптом `scripts/sync_pii_openapi.py`
- файл `contracts/pii/SYNCED_FROM.txt` фиксирует, из какого `repo@ref` была взята текущая версия

Совместимость:
- приложение ожидает `api_version=1` (см. `/health` в OpenAPI)
- при несовместимости (другая `api_version`) — показывать предупреждение и считать сервис “неподдерживаемым”
